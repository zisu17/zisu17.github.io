## 1. 개요

여러 업무 시스템의 데이터를 하나의 데이터 플랫폼으로 모으는 파이프라인에서는 **데이터 타입 변환**이 항상 따라온다. 소스는 PostgreSQL, MySQL, Oracle, SQL Server처럼 제각각이고, 적재 대상은 데이터 마트(PostgreSQL), 데이터 레이크(Iceberg), 처리 엔진(Spark)으로 나뉜다.

같은 정수라도 DBMS마다 이름이 다르다. `INT4`, `MEDIUMINT`, `NUMBER(9,0)`, `INT`가 모두 32비트 정수 계열이다. 이 대응 관계를 파이프라인 코드 곳곳에 흩어 두면 소스가 하나 늘 때마다 변환 로직을 다시 짜게 된다.

이 글에서는 이 문제를 **매핑 표 두 장**으로 정리한다. 본문의 타입 표는 실제 운영 중인 수집 파이프라인에서 쓰는 기준을 일반화한 예시다.

---

## 2. 매핑 체계: 2단계 변환

매핑은 두 단계로 나눈다.

```text
소스 DBMS 타입 → (1단계: DBMS별 정규화) → 표준 타입 → (2단계: 매핑 표) → PostgreSQL / Iceberg / Spark
```

- **1단계, DBMS별 정규화**: 각 DBMS에서 조회한 타입명을 표준 타입으로 통일한다. 예를 들어 PostgreSQL 카탈로그의 `int4`는 `INTEGER`로, MySQL의 `TINYINT(1)`은 `BOOLEAN`으로 정규화한다.
- **2단계, 표준 매핑**: 표준 타입을 기준으로 PostgreSQL, Iceberg, Spark 타입을 한 번에 결정한다.

이렇게 나누는 이유는 조합 폭발을 막기 위해서다. 소스 DBMS가 N개, 적재 대상이 M개면 직접 매핑은 N×M 조합이 필요하다. 중간에 표준 타입을 두면 소스가 늘 때 정규화 표 한 장만 추가하면 되고, 2단계 매핑 표는 그대로 재사용한다.

## 4. 타입 계열별 설계 포인트

표의 행 수는 많지만 설계 결정은 몇 가지 계열로 압축된다.

### 4.1 정수: UNSIGNED는 한 단계 큰 DECIMAL로 승격

PostgreSQL, Iceberg, Spark 모두 부호 없는 정수 타입이 없다. MySQL의 UNSIGNED 계열을 같은 크기의 부호 있는 정수에 담으면 상위 절반 범위가 오버플로된다.

그래서 UNSIGNED는 해당 범위를 통째로 담을 수 있는 **고정 소수점 타입으로 승격**한다.

| 소스 (MySQL) | 표준 매핑 |
|---|---|
| TINYINT UNSIGNED | NUMERIC(3,0) |
| SMALLINT UNSIGNED | NUMERIC(5,0) |
| INT UNSIGNED, MEDIUMINT UNSIGNED | NUMERIC(10,0) |
| BIGINT UNSIGNED | NUMERIC(20,0) |

MySQL의 `TINYINT(1)`은 관례적으로 불리언으로 쓰이므로 정수가 아니라 **BOOLEAN**으로 정규화한다.

### 4.2 실수: 정밀도에 따라 REAL과 DOUBLE PRECISION으로 분기

FLOAT라는 이름은 DBMS마다 가리키는 정밀도가 다르다. 정밀도 기준으로 32비트 계열과 64비트 계열을 나눈다.

- **REAL (32비트)**: `FLOAT(24)`, `BINARY_FLOAT` (Oracle), SQL Server의 `FLOAT(1~24)`
- **DOUBLE PRECISION (64비트)**: `FLOAT(63)`, `BINARY_DOUBLE` (Oracle), MySQL의 `DOUBLE`, SQL Server의 `FLOAT(>24)`

화폐 타입은 부동소수점 오차를 피해야 하므로 고정 소수점으로 보낸다. SQL Server의 `MONEY`는 `NUMERIC(19,4)`, `SMALLMONEY`는 `NUMERIC(10,4)`다.

### 4.3 문자열: 결국 전부 string으로 수렴

고정 길이(`CHAR`), 가변 길이(`VARCHAR`), 무제한(`TEXT`, `CLOB`) 모두 Iceberg에서는 `string`, Spark에서는 `StringType()` 하나로 수렴한다. 길이 제약은 소스 시스템의 스키마 관심사이고, 분석 플랫폼 쪽에서는 유지할 실익이 없다.

Oracle의 `LONG` 타입은 Oracle 스스로 사용 중단(deprecated)을 권고하는 타입이라 `TEXT` 계열로 흡수한다. MySQL의 `ENUM`도 코드 값 문자열로 취급해 `TEXT`로 보낸다.

### 4.4 시간: Spark에 time 타입이 없다

이 매핑 표에서 가장 주의할 지점이다. Iceberg는 `time` 타입을 지원하지만 **Spark에는 시각(time-of-day)만 담는 타입이 없다**. 그래서 `TIME`, `TIMETZ` 계열은 Iceberg에는 `time`으로 적재하더라도 Spark 스키마에서는 `StringType()`으로 다룬다.

타임존 포함 여부와 무관하게 `TIMESTAMP`, `TIMESTAMPTZ`는 모두 Spark `TimestampType()` 하나로 매핑한다. MySQL `DATETIME`, SQL Server `DATETIME2`, `SMALLDATETIME`, Parquet의 `INT96`도 같은 행으로 정규화된다.

### 4.5 반정형과 특수 타입: string으로 보존

`JSON`, `JSONB`, `XML`, `UUID`, 공간 타입(`GEOMETRY`, `GEOGRAPHY`)은 모두 문자열로 저장한다. 원본 표현을 그대로 보존하고, 해석은 조회 시점에 맡기는 방식이다.

Parquet의 중첩 타입 `MAP`, `STRUCT`도 JSON 문자열로 직렬화해 `JSONB` 행으로 정규화한다. 바이너리 계열(`BLOB`, `IMAGE`, `RAW` 등)은 전부 `BYTEA`/`binary`로 모은다.

## 6. 정리

- 이기종 소스의 타입 변환은 **DBMS별 정규화(1단계)와 표준 매핑(2단계)** 으로 나누면 소스가 늘어도 매핑 표를 재사용할 수 있다.
- UNSIGNED 정수는 부호 있는 타입에 담기지 않으므로 **범위를 커버하는 NUMERIC(p,0)으로 승격**한다.
- FLOAT 계열은 이름이 아니라 **정밀도 기준**으로 REAL(32비트)과 DOUBLE PRECISION(64비트)을 나눈다.
- 문자열, 반정형(JSON, XML), UUID, 공간 타입은 **string으로 수렴**시키고 해석은 조회 시점에 맡긴다.
- Spark에는 time 타입이 없어 **TIME 계열은 Spark에서 StringType()** 으로 다룬다. Iceberg에는 time으로 적재할 수 있다.
- 화폐 타입(MONEY, SMALLMONEY)은 부동소수점이 아니라 **고정 소수점(NUMERIC)** 으로 보낸다.
