## 1. 개요

Lakebase는 레이크하우스의 오픈 스토리지에서 실행되는 **서버리스 Postgres**다. 2025년 5월 Databricks가 약 10억 달러에 인수한 **Neon**의 스토리지·컴퓨트 분리 기술을 기반으로, 트랜잭션 데이터베이스와 데이터 레이크를 하나의 구조로 통합했다.

이 글은 Databricks 공동창업자 Reynold Xin이 제시한 Lakebase의 설계 사상을 따라가며 다음 질문에 답한다.

- OLTP 데이터베이스와 분석 시스템은 왜 항상 분리되어 있는가?
- CDC와 ETL 파이프라인 없이 운영 데이터를 실시간으로 분석할 수 있는가?
- HTAP는 왜 실패했고, LTAP는 무엇이 다른가?
- 페타바이트급 DB를 수 초 만에 복제하는 브랜칭은 어떻게 가능한가?

이 질문들은 단순히 더 빠른 DB나 더 큰 웨어하우스로 답할 수 없다. 데이터베이스가 한 서버 안에서 무엇을 함께 관리하는지, 구성요소를 어느 계층에서 분리하는지가 핵심이기 때문이다.

---

## 2. 전통적 데이터 시스템의 문제

### 2.1 OLTP와 OLAP의 분리

데이터 시스템은 전통적으로 두 영역으로 나뉘어 있다.

| 구분 | 워크로드 | 담당 시스템 |
|---|---|---|
| OLTP (온라인 트랜잭션 처리) | 주문, 결제, 회원가입처럼 짧고 빈번한 읽기·쓰기 | 서비스 DB |
| OLAP (온라인 분석 처리) | 대량 데이터를 모아 집계, 통계, 리포트 | 데이터 웨어하우스 |

문제는 이 분리가 **같은 데이터를 두 벌 이상 유지**하게 만든다는 점이다.

서비스 DB의 데이터를 분석 시스템으로 옮기기 위해 CDC(변경 데이터 캡처)와 ETL 파이프라인을 만들고, 그 파이프라인을 계속 운영해야 한다. 그 과정에서 데이터 지연과 불일치가 생긴다.

### 2.2 모놀리식 DB의 한계

전통적 데이터베이스는 한 서버 안에 두 가지를 함께 둔다.

- **WAL(Write-Ahead Log)**: 변경 내용을 데이터 파일보다 먼저 기록하는 로그. 빠른 커밋과 장애 복구의 근거다.
- **데이터 파일**: 데이터의 현재 상태. 빠른 읽기의 근거다.

이 결합이 연쇄적인 문제를 만든다.

- 내구성이 단일 머신의 생존에 의존한다.
- 읽기를 늘리려면 물리적 복제본을 만들어야 한다.
- 고가용성을 위해 중복 복제본이 필요하다.
- 분석 쿼리가 트랜잭션 트래픽과 자원을 놓고 경합한다.

Lakebase는 여기서 출발한다.

> "지금 OLTP 데이터베이스를 새로 설계한다면 어떤 모습일까?"

## 3. Lakebase의 분리 설계

### 3.1 핵심 아이디어: 컴퓨트는 데이터를 소유하지 않는다

Lakebase는 WAL과 데이터 파일을 **서버 밖 독립 서비스로 분리**한다. 그 결과 Postgres 컴퓨트는 데이터를 소유하지 않는 **상태 없는(stateless) 계층**이 된다.

데이터는 저비용 오브젝트 스토리지(예: Amazon S3)에 오픈 포맷으로 저장되고, 컴퓨트는 그 위에서 독립적이고 탄력적으로 실행된다. 컴퓨트 계층은 사실상 표준 PostgreSQL 16이라 기존 Postgres 생태계, 도구와 호환된다.

### 3.2 핵심 구성요소

**SafeKeeper (WAL 서비스)**

- 로컬 디스크 WAL 쓰기를 대체한다.
- Paxos 기반 분산 복제로 커밋을 처리한다. 디스크 flush가 아니라 쿼럼 합의로 내구성을 확보한다.
- 단일 디스크 장애나 설정 실수로 인한 탐지되지 않는(silent) 데이터 손실을 방지한다.

**PageServer (데이터 서비스)**

- 로컬 데이터 파일을 대체한다.
- WAL 스트림을 페이지로 물질화(materialize)해 오브젝트 스토리지에 저장한다. write-through 캐시처럼 동작한다.
- Postgres는 로컬 버퍼풀과 디스크 캐시로 성능을 유지한다.

**Stateless Postgres 컴퓨트**

- 데이터를 소유하지 않는 순수 계산 계층이다.
- 즉시 시작, 정지, 복제가 가능하다.

### 3.3 분리로 얻는 것

| 이점 | 내용 |
|---|---|
| 무제한 탄력 스토리지 | 오브젝트 스토리지의 내구성(99.999999999%)을 그대로 활용 |
| 서버리스 컴퓨트 | 수요에 따라 즉시 확장, 유휴 시 1초 내 0까지 축소 |
| 즉시 브랜칭·클로닝 | 페타바이트급 DB도 물리 복사가 아닌 메타데이터 연산으로 수 초 내 복제 |
| 단순한 고가용성 | 컴퓨트가 데이터를 소유하지 않아 장애 대응이 단순 |

브랜칭이 메타데이터 연산이라는 점이 중요하다. 데이터를 복사하는 것이 아니라 특정 시점을 가리키는 포인터를 만드는 것이므로, Git 브랜치처럼 격리된 실험 환경을 즉시 만들 수 있다.

## 4. LTAP: 스토리지 계층에서 통합한다

### 4.1 HTAP는 왜 실패했나

여기서 자연스러운 의문이 생긴다.

> "하나의 엔진으로 OLTP와 분석을 모두 처리하면 되지 않을까? (HTAP)"

실제로는 기대한 결과를 내지 못했다. HTAP가 정착하지 못한 이유는 세 가지다.

- 단일 엔진이 Postgres나 Spark의 발전을 따라가지 못했다.
- 신규 독점 엔진은 생태계와 도구가 부족했다.
- 하드웨어를 공유하니 두 워크로드가 서로 간섭했다.

**LTAP**(Lake Transactional/Analytical Processing)는 접근을 바꾼다.

```text
엔진은 각자 특화된 것을 그대로 쓴다.
통합은 스토리지 계층에서만 한다.
```

둘의 차이는 결국 어느 계층에서 통합하느냐로 갈린다.

| 구분 | HTAP | LTAP |
|---|---|---|
| 통합 지점 | 엔진(컴퓨트) 계층 | 스토리지 계층 |
| 방식 | 하나의 엔진이 OLTP·OLAP를 모두 처리 | 엔진은 각자 특화된 것을 쓰고 저장소만 공유 |
| 데이터 | 한 엔진 안에서 공유 | 물질화된 단일 사본을 행+컬럼 두 포맷으로 보관 |
| 대표 예 | 초기 인메모리 통합 DB | Databricks Lakebase |

HTAP는 엔진을 합치려다 성숙도·생태계·자원 경합의 벽에 막혔고, LTAP는 엔진 대신 스토리지를 합쳐 같은 목표를 달성한다.

### 4.2 작동 원리: 단일 사본, 두 포맷

핵심은 PageServer가 물질화하는 시점에 있다.

```text
쓰기
→ WAL (SafeKeeper)
→ PageServer가 페이지로 물질화
→ 이때 Postgres 행 포맷을 Parquet 컬럼 포맷으로도 트랜스코딩
→ 오브젝트 스토리지에 저장

읽기 (트랜잭션)
→ Postgres가 행 기반 캐시로 처리

읽기 (분석)
→ Spark, Databricks SQL이 컬럼 포맷을 직접 읽음
```

이 **단일 사본 하나**가 두 용도를 동시에 수행한다.

따라서 별도의 ETL 파이프라인도, CDC 복제도, 데이터 드리프트도 없다. 이것이 스토리지 계층을 직접 읽는 **Zero-CDC 분석**의 실체다.

### 4.3 기존 CDC/ETL 방식과의 비교

| 구분 | 기존 CDC/ETL | LTAP |
|---|---|---|
| 데이터 사본 | 원본 + 분석용 복제본 (2벌 이상) | 물질화된 단일 사본 공유 |
| 파이프라인 | 구축과 운영 필요 | 불필요 |
| 대상 테이블 | 선택 테이블만 복제 설정 | 모든 테이블 자동 |
| 최신성 | 배치나 복제 지연 존재 | 실시간 (LSN 기반) |
| 거버넌스 | 시스템별 개별 관리 | Unity Catalog 단일 거버넌스 |

### 4.4 해결해야 할 기술 과제

행 포맷과 컬럼 포맷을 하나의 사본으로 유지하려면 여러 기술 과제를 해결해야 한다.

**타입 보존.** NaN, 확장 타입 같은 Postgres 고유 시맨틱을 컬럼 파일의 overflow 필드에 담아 무손실 왕복(round-trip)을 보장한다.

**다중 버전(MVCC).** 모든 행 버전을 물리 heap 주소로 재구성할 수 있어, Postgres의 스냅샷 격리와 시점 복구(PITR)를 유지하면서 분석 엔진에는 일관된 스냅샷을 제공한다.

**최신성.** 분석 쿼리는 Postgres에 현재 **LSN**(Log Sequence Number, WAL의 시점 일련번호)만 조회한다. 처리 비용이 낮은 메타데이터 조회다. 그다음 오브젝트 스토리지에서 물질화 데이터를 읽고, 미커밋 변경분만 PageServer에서 가져온다.

```text
분석 쿼리 부하가 트랜잭션 워크로드에 직접 영향을 주지 않는다.
```

**커버리지.** CDC처럼 테이블을 골라 복제 설정할 필요 없이, 모든 테이블에 자동 적용된다.

## 5. 설계 원칙과 레이크하우스 통합

### 5.1 7대 설계 원칙

1. 스토리지와 컴퓨트 분리
2. 무제한 내구 스토리지 (11-nines)
3. 탄력적 서버리스 컴퓨트
4. 즉시 브랜칭과 클로닝
5. 유휴 시 비용 0 (1초 내 스케일 다운)
6. OLTP/OLAP 통합
7. 오픈 포맷, 멀티클라우드 (벤더 락인 회피)

### 5.2 레이크하우스와의 연결

- **Delta Lake**: 모든 쓰기가 레이크하우스 스토리지에 오픈 포맷으로 저장되므로, Spark와 Databricks SQL이 ETL 없이 즉시 조회한다.
- **Unity Catalog**: OLTP와 OLAP를 하나의 거버넌스 체계로 관리한다.

관리형 Postgres 서비스나 레이크하우스만으로는 구현하기 어려운 구조다. 두 워크로드를 분리된 컴퓨트에서 처리해 자원 경합을 줄인다.

### 5.3 대표 활용 사례

- **AI 에이전트**: 인스턴스를 대량으로 신속하게 생성하고, 브랜칭으로 격리된 환경에서 실험한다.
- **개발과 실험**: Git 같은 브랜치로 빠르게 반복한다.
- **동시 접속이 많은 애플리케이션, 순간 부하가 큰 워크로드**: 탄력 확장과 사용량 기반 과금으로 대응한다.
- **피처 스토어, 실시간 AI**: 트랜잭션 데이터를 지연 없이 분석과 모델 서빙에 연결한다.

## 6. 무복제(Zero-Copy) 트렌드 속의 LTAP

Lakebase의 LTAP는 독립적으로 등장한 개념이 아니다. 최근 데이터 인프라의 주요 흐름과 같은 설계 방향을 공유한다.

```text
데이터를 옮기지 말고, 있는 자리에서 읽어라.
```

| 접근 | 방식 |
|---|---|
| OBDA (온톨로지 기반 가상화) | 원본 DB를 그대로 두고 질의 시점에 SQL로 변환해 조회 |
| Zero-CDC | CDC 파이프라인 없이 스토리지 계층을 직접 읽어 분석 |
| Lakebase LTAP | ETL/CDC 없이 물질화된 단일 사본을 OLTP와 OLAP가 공유 |

셋 다 복제와 동기화 파이프라인을 없애고, **무복제·실시간**으로 조회한다는 공통 철학을 가진다.

## 7. 정리

Lakebase는 데이터베이스를 스토리지부터 다시 설계한 결과다.

| 구분 | 모놀리식 DB | Lakebase |
|---|---|---|
| 구조 | WAL과 데이터 파일을 한 서버에 결합 | WAL은 SafeKeeper, 데이터는 PageServer로 분리 |
| 컴퓨트 | 데이터를 소유 | 상태 없는 Postgres |
| 데이터 사본 | 원본 + 분석용 복제본 | 행 + 컬럼 단일 사본 |
| 분석 | 복제 파이프라인을 거쳐 별도 시스템에서 수행 | 같은 데이터를 직접 읽음 (LTAP) |

기억할 것은 세 가지다.

- 컴퓨트가 데이터를 소유하지 않으면 서버리스, 브랜칭, 스케일 투 제로를 자연스럽게 구현할 수 있다.
- HTAP는 엔진 계층을 통합하지만, LTAP는 스토리지 계층에서 통합한다.
- CDC/ETL 파이프라인은 필수 구성요소가 아니다. 스토리지 계층이 두 포맷을 함께 관리하면 별도 파이프라인이 필요하지 않다.
