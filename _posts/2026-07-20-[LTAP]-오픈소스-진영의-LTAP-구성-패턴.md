## 1. 들어가며

[1편](/data/databricks-lakebase-ltap/)에서 Databricks Lakebase가 **스토리지 계층에서 OLTP와 분석을 통합**하는 LTAP를 어떻게 구현했는지 살펴봤다. 다만 Lakebase는 관리형 상용 제품이다.

그래서 이번 편은 오픈소스만으로 LTAP를 어디까지 재현할 수 있는지 다룬다. 현재 형성된 구성 패턴들을 하나씩 살피고, 장단점과 성숙도를 비교한 뒤, 상황별 선택 기준까지 정리한다.

결론부터 말하면 이렇다.

```text
Lakebase와 동일한 기능을 스토리지 계층에 통합한 완성형 오픈소스 솔루션은 없다.
하지만 LTAP를 구현하기 위한 네 가지 구성 패턴이 뚜렷하게 형성되어 있다.
```

---

## 2. 평가 기준: LTAP 3요소

패턴을 비교하려면 기준이 필요하다. 1편에서 본 LTAP의 본질을 세 가지 요소로 분해한다.

- **논리적 단일 사본**: OLTP 행 데이터와 OLAP 컬럼 데이터를 별도로 관리하지 않는다.
- **별도 파이프라인 불필요**: 별도의 CDC/ETL 서비스를 운영하지 않는다.
- **레이크 엔진 직접 읽기**: Spark, Trino, DuckDB가 오픈 포맷으로 조회한다.

관건은 두 번째다. Debezium으로 CDC를 구성하면 나머지 두 조건은 충족할 수 있지만 파이프라인 운영 부담은 그대로 남는다. 최근 오픈소스 기술은 CDC 기능을 시스템 내부로 흡수하는 방향으로 발전했고, 기능을 담당하는 계층에 따라 다음 네 가지 패턴으로 나뉜다.

---

## 3. 왜 Neon만으로는 안 되는가

패턴을 보기 전에, 가장 흔한 오해 하나를 짚어야 한다.

> Lakebase의 엔진인 Neon이 오픈소스(Apache 2.0)라던데, Neon을 셀프호스팅하면 Lakebase가 되지 않을까?

그렇지 않다. Neon 셀프 호스팅으로 제공되는 범위는 **OLTP 계층**에 한정된다.

| Neon 오픈소스가 주는 것 | Neon 오픈소스에 없는 것 |
|---|---|
| 스토리지·컴퓨트 분리 (pageserver, safekeeper) | LTAP (스토리지의 컬럼 표현) |
| copy-on-write 브랜칭 | 분석 엔진이 읽을 Parquet |
| scale-to-zero | 관리형 컨트롤 플레인 (HA, 오토스케일) |

Neon 스토리지는 Postgres 페이지를 저장하지만 Parquet은 저장하지 않는다. 분석 계층은 별도로 구성해야 한다.

그래서 다음 단계로 Neon에 `pg_lake`(Postgres Iceberg 확장)를 결합하는 방안을 떠올리기 쉽지만, 이 구성도 구현하기 어렵다. 이유는 두 가지다.

**첫째, 계층 충돌.** pg_lake는 Postgres에 C 확장을 컴파일하고 사이드카 프로세스(`pgduck_server`)도 실행해야 한다. 반면 Neon compute는 자체 패치가 적용된 Postgres다. 두 기술의 적용 계층이 겹치므로 충돌이 발생한다. 관리형 Neon은 확장 allowlist 제약 때문에 설치할 수 없고, 셀프 호스팅 환경에서도 패치 충돌로 인해 운영 안정성이 낮다.

**둘째, 브랜칭 범위 제한.** Neon의 주요 기능인 브랜칭은 pageserver의 heap 데이터만 분기한다.

```text
Neon 브랜치 생성
→ heap + 카탈로그 메타만 분기
→ S3의 Iceberg 실제 데이터는 공유

결과: 브랜치에서 Iceberg에 데이터를 쓰면 원본 레이크 데이터에 영향을 줄 수 있다.
```

Neon을 사용하는 주요 이유가 브랜치 격리이지만 Iceberg 데이터에는 이 격리가 적용되지 않는다. 따라서 오픈소스 진영에서는 모든 기능을 Neon에 통합하기보다 역할을 분리한 네 가지 구성 패턴이 발전했다.

## 4. 패턴 1: Postgres 확장형 (Postgres 기반 레이크하우스)

첫 번째 패턴은 Postgres **확장(extension)이 CDC와 컬럼 저장을 흡수**하는 방식이다. 별도 파이프라인 서비스가 없다.

| 구현 | 방식 |
|---|---|
| `pg_lake` | 분석 테이블을 Iceberg로 **직접 생성·쓰기** (완전한 PG 트랜잭션). heap과 Iceberg를 한 SQL에서 혼합 조회. Snowflake가 Apache 2.0으로 공개 |
| `pg_mooncake` | PG 테이블의 Iceberg 컬럼 **미러**를 확장 내부 logical replication으로 자동 유지. 관리형 Neon이 공식 지원 |
| `pg_ducklake` | DuckLake 포맷 테이블을 PG에서 생성·쿼리 |
| `BemiDB` | PG 소스를 Iceberg로 동기화하는 분석 전용 리드 레플리카. 통합 구성 제공 |

실전 구성은 두 갈래로 수렴한다.

- **Neon + pg_mooncake**: OLTP의 브랜칭과 scale-to-zero를 유지하면서 Iceberg 미러를 자동으로 얻는다. 관리형 Neon에서 바로 시작할 수 있다.
- **표준 PostgreSQL + pg_lake**: 분석 테이블이 Iceberg에만 존재하므로 물리적 중복이 없는 단일 사본을 유지한다. Postgres 자체를 레이크하우스로 사용한다. 대신 Neon의 브랜칭과 scale-to-zero 기능은 사용할 수 없다.

여기서 DuckLake를 Lakebase의 오픈소스 버전으로 혼동하기 쉽지만 둘은 범주가 다르다. **DuckLake는 데이터베이스가 아니라 테이블 포맷**이다. 메타데이터는 SQL DB(Postgres 등)에, 데이터는 Parquet에 저장하는 레이크하우스 형식으로, 비교 대상은 Iceberg와 Delta Lake다. Lakebase에서 Postgres는 데이터베이스 본체이고, DuckLake에서 Postgres는 메타데이터 저장소 역할을 한다.

- **장점**: 별도 파이프라인 서비스가 필요하지 않다. PG 중심으로 구성하므로 진입 장벽이 낮고, 오픈 포맷이라 외부 엔진이 직접 읽을 수 있다.
- **한계**: 대부분 2024~2026년에 등장한 신흥 프로젝트다. 쓰기 처리량은 단일 PG 노드의 성능에 의해 제한된다.

## 5. 패턴 2: 스트림하우스 (Flink CDC + Apache Paimon)

두 번째 패턴은 CDC를 **스트림 처리 계층으로 흡수**하는 방식이다. 전통적 CDC 파이프라인과 비교하면 차이가 분명하다.

```text
전통 방식:
PostgreSQL → Debezium → Kafka → Flink → Iceberg
(4개 시스템을 각각 운영)

스트림하우스:
PostgreSQL → Flink CDC → Paimon on S3
(선언적 잡 하나. Kafka도 별도 Debezium 운영도 없음)
```

핵심은 두 가지다.

- **Flink CDC의 데이터베이스 단위 동기화**: `paimon-flink-action`의 sync-database로 테이블 자동 생성과 스키마 진화(원본 컬럼 변경 자동 반영)까지 하나의 잡으로 처리한다.
- **Paimon**: CDC upsert에 최적화된(LSM 기반) 스트리밍 레이크 포맷. Spark, Trino, StarRocks, Doris가 직접 읽고, ODS에서 DWS까지의 계층 가공도 Kafka 없이 Paimon 위에서 스트리밍으로 잇는다.

- **장점**: 오픈소스 진영에서 **대규모 프로덕션 레퍼런스가 가장 많다** (알리바바 생태계 중심). 멀티소스(여러 DB에서 한 레이크로)와 스트리밍 가공을 하나의 구성으로 처리한다.
- **한계**: Flink 클러스터와 잡 운영은 남는다. 파이프라인이 사라진 것이 아니라 한 개의 선언적 잡으로 축소된 것이다.

---

## 6. 패턴 3: 스트리밍 DB 소유형 (RisingWave)

세 번째 패턴은 CDC를 **스트리밍 데이터베이스 안으로 흡수**하는 방식이다.

```text
PostgreSQL ──(내장 postgres-cdc 커넥터)──> RisingWave
                                            ├─ Iceberg Table Engine: Iceberg를 직접 생성·관리
                                            ├─ exactly-once upsert
                                            ├─ 자동 컴팩션 + 카탈로그 호스팅
                                            └─ 실시간 구체화 뷰 서빙 (PG 와이어 호환)
[Spark / Trino / DuckDB] ──직접 읽기──> Iceberg
```

RisingWave(Apache 2.0)는 Debezium 인프라 없이 변경분을 수집하고, 2025년 출시된 Iceberg Table Engine으로 수집, 저장, 유지보수, 서빙을 한 시스템이 담당한다.

- **장점**: 오픈소스 중 **LTAP에 가장 가까운 단일 시스템 운영 환경**을 제공한다. 운영 대상이 하나다.
- **한계**: 신흥 시스템이라 성숙도가 낮고, OLTP 원본 DB는 여전히 별도로 존재한다(앱의 주 DB를 대체하지 않는다).

---

## 7. 패턴 4: 분산 HTAP (TiDB + TiFlash)

마지막 패턴은 **완전한 오픈소스(Apache 2.0) HTAP인 TiDB**를 사용하는 방식이다.

```text
TiKV (행 저장소, Raft)
  │  Raft Learner 복제: 투표 없이 로그만 받아 컬럼으로 변환
  ▼
TiFlash (컬럼 저장소)

같은 SQL 계층이 쿼리 성격에 따라 행/컬럼을 자동 라우팅
```

TiFlash는 Raft Learner이므로 OLTP에 미치는 영향이 적고, Raft index와 MVCC 검증으로 스냅샷 격리 수준의 일관된 읽기를 보장한다. 1편에서 본 Lakebase의 행 저장 + 컬럼 표현 + LSN 일관성 구조와 개념적으로 가장 유사한 성숙한 구현체다.

- **장점**: 한 시스템과 한 SQL 계층에서 강한 일관성을 제공한다. 네 가지 패턴 중 성숙도가 가장 높고 스케일아웃 OLTP까지 지원한다.
- **한계**: 컬럼 데이터가 엔진 내부 포맷이다. 레이크 엔진이 직접 읽지 못하므로 LTAP 3요소 중 오픈 포맷 직접 읽기를 충족하지 못한다. MySQL 호환이라 Postgres 전제라면 제외된다.

## 8. 종합 비교와 선택 가이드

### 8.1 3요소 기준 비교

| 구분 | 1. PG 확장형 | 2. 스트림하우스 | 3. RisingWave | 4. TiDB+TiFlash |
|---|---|---|---|---|
| 대표 구현 | pg_lake, pg_mooncake | Flink CDC + Paimon | Iceberg Table Engine | TiKV + TiFlash |
| 단일 사본 | pg_lake는 단일, 나머지 미러 | 미러 | 미러(시스템이 은닉) | 시스템 내부 단일 |
| 별도 파이프라인 불필요 | 충족 (확장 내부) | 부분 충족 (Flink 잡 1개) | 충족 (내장 CDC) | 충족 (엔진 내장) |
| 오픈포맷 직접 읽기 | 충족 | 충족 (Paimon) | 충족 (Iceberg) | **미충족** |
| 성숙도 | 신흥 (2024~) | **높음** | 신흥 | **최고** |

### 8.2 선택 가이드

| 상황 | 선택 |
|---|---|
| Postgres 중심, 중소규모, 오픈 포맷 지향 | 패턴 1 (브랜칭 중요하면 Neon+pg_mooncake, 단일 사본 중요하면 PG+pg_lake) |
| 멀티소스, 대규모, 스트리밍 가공이 함께 필요 | 패턴 2 (현재 가장 성숙한 선택지) |
| 실시간 서빙과 레이크 적재를 한 시스템으로 | 패턴 3 |
| 레이크 포맷 불필요, 스케일아웃 OLTP+분석 SQL이 목적 | 패턴 4 |

성장 경로도 자연스럽게 이어진다. 패턴 1로 작게 시작해 규모가 커지면 패턴 2로 넘어가고, 실시간 서빙 요구가 생기면 패턴 3을 병행하는 순서다.

## 9. 정리

1편의 Lakebase가 제시한 과제는 복제 파이프라인 없이 OLTP와 분석을 통합하는 것이다. 오픈소스 진영에서는 단일 완성형 제품보다 **CDC 기능을 어느 계층에 포함할 것인지**에 따라 여러 구성으로 구현하고 있다.

- 패턴 1: Postgres 확장 안으로 (pg_lake, pg_mooncake)
- 패턴 2: 스트림 잡 하나로 (Flink CDC + Paimon)
- 패턴 3: 스트리밍 DB 안으로 (RisingWave)
- 패턴 4: 분산 엔진 안으로 (TiDB + TiFlash)

기억할 것은 세 가지다.

- Neon 셀프 호스팅만으로는 LTAP를 구현할 수 없다. 제공 범위가 OLTP 계층에 한정된다.
- 완성형 오픈소스 솔루션은 없지만, 파이프라인 기능을 시스템 내부로 흡수하는 방향은 오픈소스와 관리형 서비스(Lakebase, zero-ETL)에서 공통으로 나타난다.
- 성숙도가 중요하면 패턴 2, 단순한 구성이 중요하면 패턴 1, LTAP에 가까운 운영 환경이 필요하면 패턴 3이 적합하다.
